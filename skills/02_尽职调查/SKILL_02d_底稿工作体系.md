# SKILL_02d：底稿工作体系

## 概述

DD底稿是投行尽职调查最基础也是最核心的工作档案。它不仅支撑招股说明书/申报文件的每一项陈述，更是日后监管检查、内部质控复核、法律纠纷中证明"勤勉尽责"的关键证据。

## 二、底稿命名规范

### 2.1 标准命名格式

```
{项目缩写}_{DD编号}_{文件描述}_{版本}_{日期}.{扩展名}
```

**命名示例**：
```
ABC_IPO_DD-01-01_营业执照_20240115_v3.pdf
ABC_IPO_DD-05-04_大额交易核查底稿_20240220_v2.xlsx
ABC_IPO_DD-06-08_CEO访谈纪要_20240128_v1.docx
ABC_MA_DD-02-001_标的审计报告_20240510_v1.pdf
```

**字段说明**：

| 字段 | 规则 | 示例 |
|------|------|------|
| 项目缩写 | 大写字母缩写，不超过10个字符 | ABC_IPO / ABC_MA / ABC_BOND |
| DD编号 | 与底稿目录完全一致，用连字符分隔 | DD-01-01 / DD-05-04 |
| 文件描述 | 中文描述，不超过20字，不含空格和特殊字符 | 营业执照 / 大额交易核查底稿 |
| 版本 | 见下方版本规则 | v1 / v2 / final |
| 日期 | YYYYMMDD 格式（8位数字） | 20240115 |
| 扩展名 | 小写，与文件格式一致 | .pdf / .xlsx / .docx |

### 2.2 版本号规则

| 版本标识 | 含义 | 使用场景 |
|----------|------|----------|
| v1 | 初稿 | 首次创建的文件 |
| v2, v3... | 修订稿 | 经修改但未最终定稿 |
| v{数字}-Draft | 讨论稿 | 内部讨论、尚未正式使用 |
| final | 定稿 | 经质控/内核确认，不可修改 |
| v{数字}-Annex | 附件 | 附属于主文件的支持性文件 |

**版本管理铁律**：
- ✅ 每次修改必须升级版本号（v1 → v2）
- ✅ 旧版本移至 `/archive/` 目录，不得删除
- ✅ `final` 版本设为只读，任何人不得修改
- ❌ 禁止覆盖原文件（即禁止"另存为"覆盖同名文件）

### 2.3 文件格式规范

| 文件类型 | 首选格式 | 备选格式 | 禁止格式 |
|----------|----------|----------|----------|
| 证照/批文/合同 | PDF（扫描件） | PDF（电子版） | JPG/PNG（内容模糊） |
| 分析报告/底稿 | Excel (.xlsx) | CSV | WPS格式 |
| 访谈纪要 | Word (.docx) | PDF（签署版） | 纯文本（无格式） |
| 证照照片（备用）| JPG（≥300dpi） | PNG | HEIC/RAW |

**PDF格式要求**：
- 扫描件分辨率 ≥ 300 dpi
- 文字清晰可辨，方向正确（非侧置/倒置）
- 多页文件合并为单个PDF（不得分拆为多个文件）
- 文件大小 ≤ 50MB，如超过需压缩或分拆

**命名禁用字符**（跨平台兼容要求）：
```
❌ 禁止： / \ : * ? " < > |
✅ 允许： _ - （中文/英文/数字）
```

## 三、底稿归档标准

### 3.1 电子底稿存储规范

**存储路径结构**：
```
/项目底稿（根目录）/
├── 00-项目管理/
├── 01-主体资格/
├── 02-股权结构/
├── ...
├── archive/          ← 旧版本归档目录
├── index/            ← 底稿索引文件（Excel）
└── README.md         ← 底稿目录使用说明
```

**归档标准流程**（5步法）：

```
Step 1: 命名    → 按 2.1 标准命名格式重命名文件
Step 2: 分类    → 放入对应 DD-xx 子文件夹
Step 3: 版本    → 检查版本号，旧版本移至 /archive/
Step 4: 权限    → 设置文件访问权限（见下方权限表）
Step 5: 备份    → 触发自动备份（当日完成）
```

**权限管理规范**：

| 角色 | 本模块 | 其他模块 | 归档权限 |
|------|--------|----------|----------|
| 项目负责人 | 读写 | 读写 | 全部 |
| 执行负责人 | 读写 | 读写 | 全部 |
| 项目组成员 | 读写 | 只读 | 本模块 |
| 质控/内核 | 只读 | 只读 | 无 |
| 外部审计/律师 | 指定文件夹读写 | 无 | 仅指定文件夹 |

### 3.2 纸质底稿处理规范

**原件管理**：

| 操作 | 规范 |
|------|------|
| 原件接收 | 填写《原件接收签收单》，含文件名称/数量/交接人/日期 |
| 原件保管 | 放入带锁档案柜，标注"原件—请勿带离" |
| 原件使用 | 仅限查阅，禁止带离档案室；需复印时由档案管理员操作 |
| 原件归还 | 填写《原件归还签收单》，核对文件完整性 |

**复印件管理**：
- 关键证照（营业执照、章程、批文）必须保留复印件归档
- 复印件须加盖"与原件一致"核对章 + 核对人签字 + 核对日期
- 复印件按DD编号分类装订，封面标注文件清单

**纸质底稿装订规范**：
- 使用统一封面（含项目名称、DD编号、创建日期、负责人）
- 使用彩色分页纸区分不同底稿模块（DD-00白色、DD-01蓝色、DD-02绿色...）
- 每份文件须加盖"已归档"印章
- 装订后放入专用档案盒，盒脊标注DD编号范围

### 3.3 归档时间要求

| 底稿类型 | 归档时限 | 逾期处理 |
|----------|----------|----------|
| 每次核查完成后 | 5个工作日内 | 项目例会上通报，限期3日内完成 |
| 底稿修改/更新后 | 版本更新后2个工作日内 | 系统自动提醒 + 人工催办 |
| 申报前全部底稿 | 申报前10个工作日完成 | 未完成的模块不得申报 |
| 回复反馈后补充底稿 | 反馈回复定稿后3个工作日内 | 与反馈回复同步提交质控 |

## 四、底稿复核机制

### 4.1 质控复核 Checklist

每份底稿须经质控复核，复核人在底稿封面签署意见。复核要点如下：

| 复核项目 | 检查要点 | 通过标准 |
|----------|----------|----------|
| 签字完整性 | 所有需签字位置均有有效签字 | 无遗漏签字 |
| 日期合理性 | 文件日期与核查日期逻辑一致 | 无日期倒置/矛盾 |
| 版本一致性 | 电子版与纸质版版本号一致 | 版本号完全匹配 |
| 关联逻辑性 | 同一事项在不同底稿中描述一致 | 无矛盾/冲突 |
| 数据准确性 | 底稿数据与审计报告/合同一致 | 勾稽关系正确 |
| 引用完整性 | 招股书陈述均有对应底稿索引 | 索引完整无遗漏 |
| 文件有效性 | 证照/批文在有效期内 | 无过期文件 |
| 扫描质量 | 文字清晰/方向正确/无缺页 | 满足阅读/打印要求 |

**复核结果处理**：
- ✅ **通过**：复核人在底稿封面签署"已复核，{日期}，{姓名}"，底稿可进入下一环节
- ⚠️ **有条件通过**：列出需补充事项，限期3日内完成整改
- ❌ **不通过**：退回执行负责人，重新整理后再次提交复核

### 4.2 常见质控问题清单（15项高频问题）

| 序号 | 问题类型 | 具体表现 | 整改要求 |
|------|----------|----------|----------|
| 1 | 签字缺失 | 访谈纪要缺被访谈人签字 | 补充签字或书面确认函 |
| 2 | 日期逻辑错误 | 合同签署日期早于营业执照日期 | 核实真实日期，更正底稿 |
| 3 | 版本混乱 | 电子版v3，纸质版v2 | 统一版本，旧版本归档 |
| 4 | 数据前后不一致 | 底稿中营收数据与审计报告不一致 | 追溯源头，统一数据口径 |
| 5 | 引用法规已废止 | 引用已废止的《证券法》条款 | 更新为现行有效条款 |
| 6 | 证照过期未更新 | 财务报表审计截止日后证照已过期 | 补充最新证照或延期证明 |
| 7 | 缺少索引标注 | 招股书陈述无对应底稿编号 | 补充底稿或在底稿中标注索引 |
| 8 | 底稿与函证不符 | 函证回函金额与底稿记录不一致 | 核实差异原因，调整底稿 |
| 9 | 遗漏签字页 | 合同/决议文件缺最后一页（签字页） | 补充完整文件 |
| 10 | 扫描件缺页 | PDF文件缺失中间页面 | 重新扫描，确保完整 |
| 11 | 关联方识别不全 | 底稿未涵盖全部关联方 | 重新核查企查查/工商档案 |
| 12 | 复印件未盖核对章 | 关键文件复印件无"与原件一致"章 | 补盖核对章+签字+日期 |
| 13 | 底稿描述与事实不符 | 底稿记录与实际情况有出入 | 实地/独立核查，更正记录 |
| 14 | 缺少分析性底稿 | 仅有原始资料，无分析判断记录 | 补充分析底稿，记录判断依据 |
| 15 | 时间逻辑错误 | 董事会决议日期早于会议通知日期 | 核实真实程序，更正底稿记录 |

## 五、监管检查应对

### 5.1 证监会现场检查底稿清单（前10项必查项）

> 💡 以下10项为底稿检查中最高频调阅的文件，须确保完整性、真实性、可解释性。

| 序号 | 必查底稿 | 检查要点 | 准备要求 |
|------|----------|----------|----------|
| 1 | 审计报告及全套工作底稿 | 审计调整分录、审计策略、关键审计事项 | 提前与会计师沟通，确保口径一致 |
| 2 | 银行流水核查底稿 | 单笔大额/异常资金往来、与收入/成本的匹配 | 准备资金往来说明备忘录 |
| 3 | 收入确认穿行测试底稿 | 从合同到收款的全流程、截止性测试 | 准备代表性样本的完整证据链 |
| 4 | 关联方及关联交易底稿 | 关联方识别过程、关联交易定价公允性分析 | 准备关联交易必要性说明 |
| 5 | 存货盘点底稿 | 盘点计划、盘点表、差异说明、监盘记录 | 准备盘点照片/视频作为补充证据 |
| 6 | 重大合同原件及核查记录 | 合同真实性、条款与披露一致性 | 准备合同台账及主要条款摘要 |
| 7 | 不动产权证及抵押情况 | 权证真实性、抵押/查封查询记录 | 准备不动产查档记录 |
| 8 | 知识产权清单及法律状态 | 专利/商标有效性、年费缴纳凭证 | 准备CNIPA官方检索报告 |
| 9 | 诉讼/仲裁/处罚核查底稿 | 核查范围、方法、结论、未披露事项的说明 | 准备未决诉讼风险管理预案 |
| 10 | 内控鉴证报告及整改底稿 | 内控缺陷识别、整改措施、整改验收 | 准备整改前后对比说明 |

### 5.2 现场检查应答要点

**应答基本原则**：

```
原则1：实事求是
  → 已知事项：正面回答，提供底稿证据
  → 未知事项：如实说明"需进一步核查"，不猜测/不编造

原则2：逻辑自洽
  → 答复内容与底稿记录一致
  → 答复内容与招股书披露一致
  → 答复内容与会计师/律师意见一致

原则3：留痕管理
  → 每次问答均做书面记录
  → 提供的补充底稿须签收登记
  → 口头答复后须补充书面说明
```

**底稿逻辑链展示方法**：

当检查人员要求展示某事项的核查过程时，按以下顺序呈现：

```
Step 1: 招股书陈述    → 指出招股书第X节第X页的具体陈述
Step 2: 底稿索引      → 展示对应的底稿编号和文件名称
Step 3: 核查方法      → 说明采用了何种核查程序（访谈/函证/盘点/检索）
Step 4: 核查证据      → 展示具体证据（合同/函证回函/查询结果）
Step 5: 核查结论      → 说明核查结论及判断依据
```

**常见问题应答模板**：

| 检查人员问题 | 应答思路 | 需准备的底稿 |
|-------------|---------|-------------|
| "请说明收入确认政策的合理性" | 1.描述政策 2.对比同行 3.展示穿行测试 | 收入确认政策底稿+穿行测试+可比公司对比 |
| "关联方识别是否完整" | 1.说明识别方法 2.列出核查范围 3.披露豁免说明 | 关联方清单+核查方法说明+企查查检索记录 |
| "存货盘点的真实性如何保证" | 1.盘点计划 2.监盘记录 3.差异处理 4.盘点照片 | 盘点计划+监盘记录+差异说明+现场照片 |
| "诉讼事项是否充分披露" | 1.核查范围 2.未披露事项说明 3.律师意见 | 诉讼核查底稿+律师尽调报告+管理层声明书 |

### 5.3 检查后整改报告撰写要点

整改报告是检查结束后向监管部门提交的正式文件，须体现诚恳态度和系统整改。

**报告结构（5部分）**：

```
第一部分：整改工作组织情况
  → 整改责任人、整改时间表、整改措施概述

第二部分：核查发现问题的整改情况（逐条回复）
  → 问题描述（引用检查人员原话）
  → 原因分析（客观分析，不推卸责任）
  → 整改措施（具体、可验证）
  → 整改结果（已完成的提供证据，进行中的说明计划）
  → 责任人及完成时间

第三部分：底稿补充及完善情况
  → 补充底稿清单（含DD编号、文件名称、补充日期）
  → 底稿质量提升措施

第四部分：内部控制完善措施
  → 针对检查发现的内控缺陷，说明完善措施

第五部分：结语
  → 承诺持续规范，恳请监管部门指导
```

**整改报告撰写注意事项**：
- ✅ 态度诚恳，不推卸责任
- ✅ 整改措施具体可验证（避免"加强管理"等空话）
- ✅ 附上整改证据（补充底稿、修订文件、新的核查记录）
- ❌ 避免与检查人员争论问题定性
- ❌ 避免整改措施空洞无物

## 六、底稿工作常见错误库

### 6.1 格式错误

| 错误类型 | 具体表现 | 后果 | 预防措施 |
|----------|----------|------|----------|
| 命名不规范 | 文件名含空格/特殊字符/中文乱码 | 跨平台打开失败、系统无法识别 | 使用标准命名格式，禁止特殊字符 |
| 缺少日期 | 文件名或底稿内无日期标注 | 无法判断文件时效、版本混乱 | 文件名必须含YYYYMMDD日期 |
| 版本混乱 | 同一文件存在v1/v2/final多个版本且内容不同 | 质控使用错误版本、申报文件错误 | 严格版本管理，旧版本立即归档 |
| 文件格式不统一 | 同一类底稿有的是PDF、有的是照片 | 查阅效率低、归档不规范 | 制定格式规范并强制执行 |
| 扫描方向错误 | 扫描件侧置或倒置 | 查阅不便、显得不专业 | 扫描后预览检查，确保正向 |

### 6.2 内容错误

| 错误类型 | 具体表现 | 后果 | 预防措施 |
|----------|----------|------|----------|
| 数据前后不一致 | 同一数据在底稿A和底稿B中不同 | 监管质疑勤勉尽责、招股书需修改 | 建立数据字典，统一数据口径 |
| 引用已废止法规 | 引用已修订/废止的法律条款 | 招股书法律效力受质疑 | 建立法规版本台账，定期更新 |
| 遗漏签字页 | 合同/决议缺失签字页 | 文件法律效力不足、底稿不完整 | 接收文件时立即检查完整性 |
| 日期逻辑错误 | 决议日期早于通知日期 | 程序合规性受质疑 | 建立日期逻辑检查清单 |
| 描述与事实不符 | 底稿记录与实际情况有出入 | 监管认定为未勤勉尽责 | 每次记录均与原始文件核对 |

### 6.3 完整性错误

| 错误类型 | 具体表现 | 后果 | 预防措施 |
|----------|----------|------|----------|
| 缺附件 | 主文件引用了附件但附件未归档 | 底稿证据链不完整 | 接收文件时核对引用关系 |
| 缺索引标注 | 招股书陈述无对应底稿编号 | 无法证明陈述有底稿支撑 | 招股书撰写时同步标注底稿索引 |
| 缺关联引用 | 底稿A提及底稿B但未标注 | 底稿之间无法交叉验证 | 建立底稿交叉引用表 |
| 缺更新版本 | 证照/批文已更新但底稿未更新 | 使用过期文件、披露内容错误 | 建立文件有效期台账，到期前提醒 |
| 缺核查范围说明 | 底稿未说明核查了哪些范围 | 无法判断核查是否充分 | 每份底稿须含"核查范围"说明段 |

## 七、与债券尽调底稿的衔接

本项目底稿体系主要面向IPO/再融资/并购重组项目。对于债券类项目，底稿体系有专门规范：

- **债券尽调底稿规范** → 委托 `bond-due-diligence` 技能（`~/.qclaw/skills/bond-due-diligence/SKILL.md`）
- **债券项目全流程执行** → 委托 `ib-bond-executor` 技能（`~/.qclaw/skills/ib-bond-executor/SKILL.md`）

债券底稿与股权底稿的主要差异：

| 差异维度 | 股权底稿 | 债券底稿 |
|----------|----------|----------|
| 核心关注 | 发行条件/持续经营能力 | 偿债能力/信用水平 |
| 底稿重点 | 历史沿革/业务/财务全景 | 有息负债/现金流/增信措施 |
| 评级底稿 | 不涉及 | 必须包含评级机构尽调底稿 |
| 募集资金 | 用途可行性分析 | 用途合规性+偿债保障 |
| 增信措施 | 不涉及 | 担保方尽调+抵押物评估 |
