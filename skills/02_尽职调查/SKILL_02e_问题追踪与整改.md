# SKILL_02e：问题追踪与整改

## 概述

问题追踪系统是DD阶段的管理中枢，将散落各维度的发现问题统一管理、分级处理、追踪闭环。核心目标：**确保申报前所有Red问题清零、所有Amber问题有明确的解决方案**。

---

## 一、RAG分类体系

### 1.1 三级分类标准

| 等级 | 颜色 | 图标 | 定义 | 处理要求 | 最晚完成节点 |
|------|------|------|------|----------|-------------|
| 🔴 Red | 红色 | 🔴 | 实质性障碍。如不解决将导致：不符合发行/上市条件、被否决、构成重大违法 | 必须彻底解决后方可申报 | 内核前 |
| 🟡 Amber | 黄色 | 🟡 | 需充分披露和解释的风险或瑕疵。不影响发行条件，但需在申报文件中如实披露并在答反馈时充分论证 | 确定披露方案+完善支持性证据 | 申报前 |
| 🟢 Green | 绿色 | 🟢 | 常规管理事项。不会实质性影响项目推进，通过常规措施可管理 | 记录在案并归档 | 持续管理 |

### 1.2 RAG升级与降级规则

```
Green → Amber 升级条件：
  - 问题在2次DD循环中反复出现无法关闭
  - 新发现证据表明问题影响可能大于预期
  
Amber → Red 升级条件：
  - 整改无效或问题实质恶化
  - 监管政策变化使原可容忍的问题变为实质性障碍
  - 发现关键证据缺失或存在不实陈述

Red → Amber 降级条件：
  - 整改措施完成并经第三方验证
  - 取得监管部门的明确批复/豁免
  - 经内核委员会审议确认不再构成实质性障碍
```

## 二、问题ID格式

```
{DD}-{阶段}-{类别}-{序号}

DD = Due Diligence
阶段: L(法律) / F(财务) / B(业务)
类别: R(Red) / A(Amber) / G(Green)
序号: 4位数字

示例：
DD-L-R-0001  → 法律DD第1号Red问题
DD-F-A-0012  → 财务DD第12号Amber问题
DD-B-G-0005  → 业务DD第5号Green问题
```

| 字段 | 说明 | 示例 |
|------|------|------|
| 问题ID | 唯一编号 | DD-L-R-0003 |
| 发现日期 | 首次识别日期 | 2024-01-20 |

## 四、升级规则

### 4.1 时间驱动的升级

| 等级 | 未解决持续时间 | 升级动作 |
|------|---------------|----------|
| 🔴 Red | >1周 | 升级至项目负责人+质控负责人 |
| 🔴 Red | >2周 | 升级至部门负责人+内核 |
| 🔴 Red | >4周 | 升级至公司高管层+重新评估项目可行性 |
| 🟡 Amber | >2周 | 升级至项目负责人 |
| 🟡 Amber | >4周 | 升级至质控负责人+内核 |
| 🟢 Green | >8周 | 提醒关闭或升级至Amber评估 |

### 4.2 事件驱动的升级

| 触发事件 | 升级动作 |
|----------|----------|
| 问题影响范围扩大（>原评估2倍） | 重新评估等级+立即升级 |
| 监管政策变化使问题严重性增加 | 重新评估等级 |
| 客户明确拒绝整改 | 立即升级至高管层 |
| 发现新证据与原陈述矛盾 | 立即升级至质控/内核 |
| 整改实施中发现无法按原方案进行 | 升级至项目负责人，重新制定方案 |

### 4.3 升级报告

每次升级需提交书面升级报告，包含：

1. 问题ID和问题描述
2. 原定整改方案和截至升级时的进展
3. 无法在原时限内解决的原因
4. 新整改方案建议或替代方案
5. 对项目时间表和里程碑的影响评估
6. 需要决策的事项

## 五、关闭标准

### 5.1 各等级关闭条件

| 等级 | 关闭条件 |
|------|----------|
| 🔴 Red | ①整改措施全部执行完毕 ②整改效果经第三方验证（律师/审计师/评估师） ③质控/内核委员会确认不再构成实质性障碍 ④相关文件已归档至底稿 |
| 🟡 Amber | ①在招股书/申报文件中的披露方案已确定并定稿 ②支持性文件已收集完毕 ③质控/内核已确认披露充分性 ④相关底稿已归档 |
| 🟢 Green | ①常规管理措施执行完毕 ②在DD底稿中记录 ③定期复核确认不升级 |

### 5.2 关闭审批流

```
项目组成员  →  提出关闭申请（附支持文件）
      ↓
执行负责人  →  初步审核关闭条件满足情况
      ↓
质控负责人  →  复核（Red/Amber必须经质控）
      ↓
内核        →  确认（Red必须经内核）
      ↓
关闭登记    →  问题追踪表中标注Closed+关闭日期
```

## 六、Red→Amber→Green 流转工作流

```
┌──────────────────────────────────────────────────┐
│                   问题发现                         │
│                    ↓                              │
│              初步定级(R/A/G)                       │
│                    ↓                              │
│  ┌─────────┬─────────┬─────────┐                 │
│  │  🔴 Red  │ 🟡 Amber│ 🟢 Green│                 │
│  └────┬────┘ └────┬────┘ └────┬────┘              │
│       ↓           ↓           ↓                   │
│  制定整改方案  制定披露方案  记录归档               │
│       ↓           ↓           ↓                   │
│  执行整改    完善支持证据  定期复核                 │
│       ↓           ↓                               │
│  整改验证    质控/内核确认                          │
│       ↓           ↓                               │
│  降为Amber    关闭 ──────────┐                    │
│       ↓                      ↓                    │
│  确定披露方案              关闭                    │
│       ↓                                           │
│  完善支持证据                                     │
│       ↓                                           │
│  关闭                                            │
└──────────────────────────────────────────────────┘

Red→Amber: 实质问题解决后降到Amber（仍需披露）
Amber: 披露方案确定后直接关闭
Green: 直接关闭（仅需记录）
```

**关键原则：Red问题整改后，不是直接关闭，而是先降为Amber（因为曾经存在的瑕疵仍需在申报文件中对历史情况进行说明和披露）。**

## 七、周度问题追踪会

### 7.1 会议议程

1. 新发现问题汇总及定级（5分钟）
2. Red问题逐个过进展（15分钟）
3. Amber问题逐个过进展（10分钟）
4. 超时/需升级问题讨论（10分钟）
5. 问题关闭确认（5分钟）

### 7.2 会议产出

- 更新后的问题追踪表
- 新增问题的整改方案
- 升级问题和决策记录
- 下周重点工作安排
