# META-15: 跨产品线协同工作法

> 定位：打破产品线壁垒，实现资源复用、方案择优、团队协同的系统性方法论。

## 一、为什么需要跨产品线协同？

ib-execution 覆盖 IPO/债券/并购三条产品线。在实务中，这三条线不是孤立的：

- 同一客户可能同时有股权和债券融资需求
- 债券项目可能演变为重组项目
- 尽调成果可以在三条线间复用
- 团队资源存在竞合关系

**协同的目标**：提升效率（复用）、降低风险（比选）、优化资源配置。

---

## 二、项目启动期的三线检查表

> 承接新客户时，同时评估三条产品线的可行性，不错过任何机会。

### 2.1 三线并行检查表

| 检查维度 | IPO条件 | 发债资格 | 重组可行性 |
|----------|---------|----------|------------|
| **主体资格** | 依法设立且持续经营3年以上 | 依法设立的非金融企业 | 上市公司/非上市公众公司 |
| **财务标准** | 最近3年净利润累计≥3000万(主板) | 近3年平均可分配利润≥债券一年利息 | 标的资产符合重组条件 |
| **合规要求** | 无重大违法违规 | 无重大违法违规 | 符合产业政策 |
| **独立性** | 业务/资产/人员/财务/机构独立 | 不要求完全独立 | 注入资产需独立可核算 |
| **特殊要求** | 板块定位匹配 | 信用评级 | 交易对价公允 |

### 2.2 三线评估结论汇总

```
┌──────────────────────────────────────────────────────────────┐
│  项目启动三线评估                                            │
│                                                              │
│  ┌─────────────────┐                                        │
│  │ 客户A 评估结果   │                                        │
│  │                 │                                        │
│  │ IPO条件： ✅    │  主体合格，业绩达标，但板块定位需讨论    │
│  │ 发债资格： ✅    │  财务指标良好，评级已获AA+              │
│  │ 重组可行性： ⚠️  │  标的需进一步评估                       │
│  │                 │                                        │
│  │ 推荐路径：债券（快速）+ IPO（中长期）                     │
│  └─────────────────┘                                        │
└──────────────────────────────────────────────────────────────┘
```

### 2.3 三线评估快速决策表

| IPO条件 | 发债资格 | 推荐策略 |
|---------|----------|----------|
| ✅ | ✅ | 优先债券（快获现金流）+ 同步推进IPO |
| ✅ | ❌ | 集中资源推IPO |
| ❌ | ✅ | 集中资源推债券 |
| ⚠️（培育期） | ✅ | 债券先行，同步培育IPO条件 |
| ❌ | ❌ | 重新评估客户价值，或转重组路径 |

## 三、尽调成果复用矩阵

> 尽调是最大成本的环节，最大化跨产品线成果复用。

### 3.1 尽调成果复用矩阵

| 尽调类型 | IPO | 债券 | 并购 | 复用率 | 差异点 |
|----------|-----|------|------|--------|--------|
| **法律DD**（主体资格） | ✅ | ✅ | ✅ | **95%** | 历史沿革三线通用 |
| **法律DD**（资产权属） | ✅ | ✅ | ✅ | **90%** | 无形资产需专项 |
| **法律DD**（重大合同） | ✅ | ✅ | ✅ | **85%** | 债券侧重担保合同 |
| **财务DD**（收入核实） | ✅ | ✅ | ⚠️ | **80%** | 并购侧重点不同 |
| **财务DD**（关联交易） | ✅ | ✅ | ✅ | **90%** | 三线通用 |
| **财务DD**（偿债分析） | ⚠️ | ✅ | ⚠️ | **60%** | 债券深度要求最高 |
| **业务DD**（行业分析） | ✅ | ✅ | ✅ | **85%** | IPO最深入 |
| **业务DD**（竞争力） | ✅ | ⚠️ | ✅ | **70%** | 债券侧重点不同 |
| **业务DD**（客户供应商）| ✅ | ⚠️ | ✅ | **75%** | IPO和并购更关注 |

### 3.2 复用实现方法

**方法1：统一尽调底稿体系**

```
尽调底稿 = 通用底稿 + IPO专项底稿 + 债券专项底稿 + 并购专项底稿

通用底稿（所有线使用）：
├── 公司基本证照（营业执照、章程、股东名册）
├── 历史沿革文件
├── 财务报表及审计报告
├── 重大合同清单及主要合同
├── 关联方及关联交易清单
├── 资产权属证明
├── 税务合规证明
└── 诉讼仲裁情况
```

**方法2：交叉引用而非重复制作**

将一份尽调发现文档化，其他产品线通过交叉引用使用：

> "关于公司历史沿革，详见[IPO尽调报告第三章]"
> "关于关联交易的核查，详见[统一尽调底稿D-03]"

### 3.3 尽调复用效率提升测算

| 场景 | 独立尽调耗时 | 复用后耗时 | 节省时间 |
|------|-------------|-----------|----------|
| IPO后追加债券 | 8-12周 | 2-4周 | 50-70% |
| 债券后追加IPO | 16-24周 | 10-14周 | 35-40% |
| 三线同时启动 | 32-48周 | 20-28周 | 30-40% |

## 四、方案比选逻辑

> 同一客户面临多种融资路径时，如何进行科学比选？

### 4.1 方案比选五维评分模型

| 评估维度 | 权重 | IPO | 债券 | 借壳 | 并购整合 |
|----------|------|-----|------|------|----------|
| **融资效率**（速度） | 25% | 2-4年 | 6-18月 | 1-2年 | 1-2年 |
| **融资规模** | 20% | 大 | 中-大 | 中 | 取决于标的 |
| **成本控制** | 20% | 中等承销费率 | 利差成本 | 借壳成本高 | 对价支付 |
| **品牌效应** | 15% | 极高 | 中等 | 较高 | 取决于方案 |
| **灵活性** | 20% | 低（锁定期） | 高（多品种） | 中等 | 中等 |

**评分公式**：综合得分 = Σ(维度得分 × 权重)

### 4.2 比选决策树

```
                        ┌─────────────────┐
                        │  客户融资需求    │
                        └────────┬────────┘
                                 │
                    是否有上市资质？
                    ┌───────┴────────┐
                    │ YES            │ NO
                    ▼                ▼
           ┌───────────────┐  ┌──────────────┐
           │ 选择上市路径？ │  │ 债券/借壳？  │
           └───┬───────┬───┘  └──┬───────┬───┘
               │       │         │       │
          IPO/重组  债券先行   债券   借壳上市
               │       │
           ┌───▼───┐   │
           │IPO vs │←──┘
           │ 借壳  │
           └───────┘
```

### 4.3 典型比选场景

**场景1：成熟制造业公司，净利润3亿，AA+评级**

| 方案 | 分析 | 时限 | 规模 |
|------|------|------|------|
| IPO | 板块定位匹配、业绩达标、最佳长期方案 | 1-2年 | 10-15亿 |
| 债券 | 评级优良、当前即可启动 | 3-6月 | 5-10亿 |
| **推荐** | **债券先行（解决短期资金）+ IPO同步筹备（长期目标）** | | |

**场景2：高成长科技公司，亏损，尚未评级**

| 方案 | 分析 | 时限 | 规模 |
|------|------|------|------|
| IPO（科创板） | 亏损可接受，科创属性需论证 | 1-2年 | 较大 |
| 债券 | 亏损+无评级，发行困难 | — | — |
| 并购被收购 | 可作为退出路径 | 6-12月 | 取决于估值 |
| **推荐** | **IPO（科创板）为主，并购为备选** | | |

**场景3：国有企业，资产规模大，有重组需求**

| 方案 | 分析 | 时限 | 规模 |
|------|------|------|------|
| 整体上市 | 最彻底方案 | 2-3年 | 很大 |
| 资产注入 | 部分资产证券化 | 1-2年 | 中 |
| 债券 | 惯常融资渠道 | 3-6月 | 灵活 |
| **推荐** | **债券持续滚动 + 资产注入分步推进** | | |

## 六、混合项目执行策略

> 当客户同时开展"股+债"或"重组+再融资"等混合项目时的执行策略。

### 6.1 "股+债"联动策略

```
时间轴:
─────────────────────────────────────────────────────▶
│ ← 债券发行（3-6月）→ │ ← IPO（同时推进）→            │
│                       │                              │
│ 债券专项尽调          │ 通用尽调成果复用              │
│ 信用评级              │ 减少IPO尽调时间               │
│ 债券申报注册          │ IPO同步申报准备               │
│ 债券发行              │ (确保债券不影响IPO定价)        │
│                       │                              │
│   结果：债券融资到位 → 支撑IPO期间业务发展 → IPO申报  │
```

**关键注意点**：
- 债券发行后资产负债率变化 → 重新测算IPO募投
- 确保债券募资用途不冲突
- 协调两个项目的申报节奏

### 6.2 "重组+再融资"联动策略

```
时间轴:
───────────────────────────────────────────────────▶
│ ← 重组准备 → │ ← 重组实施 → │ ← 整合期 →          │
│              │              │                     │
│              │ ← 配套再融资（同期推进）→            │
│              │              │                     │
│  关键：重组方案中嵌入配套融资需求                   │
│  重组完成 = 上市公司体量提升 → 再融资更有利        │
```

**风险控制要点**：

| 风险点 | 控制措施 |
|--------|----------|
| 重组失败导致再融资延期 | 准备独立融资的备选方案 |
| 重组后业绩不达预期 | 业绩承诺+补偿条款 |
| 资金链断裂 | 银行过桥融资 |

### 6.3 "债券+ABS"联动策略

> 对于有合适资产基础的企业，在债券融资的同时探索ABS

```
债券融资（主体信用） + ABS融资（资产信用）

适用场景：
  - 企业有优质应收款/租赁债权/供应链资产
  - 主体信用和资产信用形成互补
  - 可有效降低综合融资成本
```

## 七、跨产品线协同的常见陷阱

| 陷阱 | 表现 | 规避方法 |
|------|------|----------|
| **"一套文件走天下"** | 将IPO尽调报告直接用于债券 | 各线有专门要求，必须专项补充 |
| **"团队混编导致责任不清"** | 谁对每条线的进度负责不清楚 | 明确每条线有独立负责人 |
| **"时间冲突导致资源空转"** | 两条线同时到关键节点抢人 | 大时间表统筹规划 |
| **"信息隔离不充分"** | 债券组碰触IPO内幕信息 | 建立信息隔离墙 |
| **"利益冲突被忽略"** | 客户的两条线实际利益不一致 | 每个项目独立评估利益冲突 |

## 八、使用指南

### 8.1 何时使用本工作法

- 新客户接触 → 先用"三线检查表"评估全维度机会
- 已有单一产品线项目 → 检查是否存在跨线机会
- 尽调完成后 → 归档成果供其他线复用
- 项目论证会 → 用"方案比选"做多路径比较
