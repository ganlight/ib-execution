# META-00: IB方法论总纲 — 好东西都是总结出来的

> 定位：所有 META 技能的元技能，贯穿冷启动、迭代、反馈回复、质量控制所有工序。

## 一、核心命题

**投行承做的东西都是总结出来的，不是设计出来的。**

三年经验的承做 vs 一年经验的承做——差距不在于"知道多少法规"，而在于"积累了多少模式"。本技能体系的目标是将这些模式显性化、可查询、可复用。

## 二、OTF（即时总结）— IB版

不等项目结束才总结，而是发现模式的那一刻立即记录。

### IB承做中的触发时机

| 时机 | 动作 | 产出 |
|------|------|------|
| 每次反馈回复完成 | 5分钟：这个问询属于哪个类？回复框架是否可复用？ | 追加模式到反馈库 |
| 每个项目结束 | 30分钟：本项目的新发现、新套路 | 更新 META-03 |
| 每季度 | 半天：跨项目规律提炼 | 升级 META 层 |

### 实操案例

```
第1个项目(科创板IPO): 问询"收入确认" → 手写回复(4小时)
→ OTF: 立即总结收入确认回复框架
第3个项目: 问询"收入确认" → 套用框架(0.5小时) ✓
第5个项目: 框架已沉淀到技能 → 直接生成初稿(5分钟) ✓
```

## 三、JIT（及时交付）— IB版

不追求文件一次完美，而是及时交付最小可用版本，快速迭代。

### 招股书迭代模型

```
P0(初稿 3天): 快速填充框架，允许大量[TBD]
P1(内核稿 1周): 消灭TBD，内核会反馈 → 修正
P2(申报稿 1周): 交叉校验 + Lint，提交申报
P3-PN(反馈稿): 每轮问询后迭代修改
```

### 核心原则

- **广度优先**: 先快速覆盖130页框架，再逐页打磨
- **模式驱动**: 每轮修改的核心产出是错误模式，而非修正本身
- **指数收敛**: 修改量应每轮递减（50+→20→8→3→0）

## 四、Bootstrap（知识增殖）

```
第1个项目: 全程摸索，大量试错
第3个项目: 积累20+反馈回复套路 → 工作提速2倍
第5个项目: 套路库覆盖80%场景 → 新人上手速度提升3倍
第10个项目: 技能体系自运转 → 人工只需决策和审核
```
