# 00-META-04 柳叶刀混合工作法（Scalpel Method）

> **核心思想**：在IB投行工作场景中，不同的任务需要不同的工具。如同外科手术中针对不同组织选用不同刀具，我们需要在 Python脚本、LLM、模板引擎、人工审阅之间精准切换，达到成本与质量的最优平衡。
> **核心原则**：用最便宜的方案完成每个子任务，而非用一个方案解决所有问题。

---

## 一、四象限模型

### 1.1 象限定义

以两个维度划分所有IB工作：

```
                结构化 Structured
                       │
        象限II        │        象限I
      模板驱动型      │      脚本/规则型
        │            │            │
  需要判断 ←─────────┼──────────→ 规则明确
  Needs Judgment      │       Rule-Based
        │            │            │
        象限III       │        象限IV
      人工主导型      │      LLM辅助型
        │            │
                非结构化 Unstructured
```

### 1.2 四象限工作分配策略

| 象限 | 特征 | 典型任务 | 推荐方案 | 准确率 | 成本 |
|------|------|---------|---------|--------|------|
| **I** - 规则明确 × 结构化 | 有明确规则、数据规律 | 数据搬运、计算、格式化检查 | 🔧 Python脚本自动 | 100% | $0 |
| **II** - 需要判断 × 结构化 | 有框架、但需专业判断 | 可比分析、行业分析、风险分类 | 🤖 LLM初稿 + 👤 人工精修 | 90% | 低→中 |
| **III** - 需要判断 × 非结构化 | 无模板、依赖经验创造 | 投资故事、核心亮点、发行方案 | 👤 人工主导 + 🤖 LLM辅助 | 95% | 高 |
| **IV** - 规则明确 × 非结构化 | 有参考但需灵活适配 | 反馈回复初稿、核查说明撰写 | 📋 模板填充 + 🤖 LLM补充 | 88% | 低 |

## 二、决策树：拿到任务如何选工具

### 2.1 决策流程图

```
收到任务
    │
    ├─ Step 1: 这是数据搬运/数值计算吗？
    │   └─ YES → 【Python脚本自动执行】→ 输出结果
    │       示例：计算应收账款迁徙率、验证财务报表勾稽关系
    │
    ├─ Step 2: 这是格式/合规性检查吗？
    │   └─ YES → 【Lint规则引擎/RPA脚本】→ 输出问题清单
    │       示例：招股书交叉引用校验、页码格式统一检查
    │
    ├─ Step 3: 是否有成熟的模板/框架？
    │   ├─ YES + 需要专业判断 → 【LLM初稿 + 人工精修】
    │   │   示例：行业分析（有分析框架）、可比公司选择论证
    │   │
    │   └─ YES + 主要靠填充 → 【模板填充 + LLM补充辅助句】
    │       示例：关联方清单格式整理、历史沿革时间线
    │
    ├─ Step 4: 是否完全需要创造力+行业理解？
    │   └─ YES → 【人工主导 + LLM辅助（提供素材/初稿/灵感）】
    │       示例：确定投资故事主线、撰写核心竞争优势
    │
    └─ Step 5: 是否涉及合规判断（有法规依据）？
        └─ YES → 【LLM + 法规检索 + 人工终审】
            示例：判断某收入确认方法是否符合会计准则
```

### 2.2 混合工作法三段式

```markdown
几乎所有任务都可以分解为:

【第一刀】数据处理/格式化 → 脚本自动
【第二刀】内容起草/分析 → LLM Draft
【第三刀】质量把关/决策 → 人工Review
```

## 三、工具矩阵：成本-质量-速度

### 3.1 核心指标对比

| 工具 | 单次成本 | 准确率 | 速度 | 适用条件 | 不适用场景 |
|------|---------|--------|------|---------|-----------|
| Python脚本 | $0 | 100% | 极快(秒级) | 规则明确、可编程 | 需要语义理解 |
| Lint规则 | $0 | 95% | 极快 | 格式/一致性 | 内容质量判断 |
| 模板填充 | $0 | 85% | 快 | 有高质量模板 | 非标内容 |
| LLM | ~$0.01-0.10 | 85% | 快(分钟级) | 结构化任务+非结构化文本生成 | 纯计算、合规终审 |
| LLM+RAG | ~$0.02-0.15 | 88% | 中等 | 需要引用法规/案例 | 时效性极强的法规 |
| 人工初级 | $20-50/h | 90% | 慢 | 模板+简单判断 | 高级决策 |
| 人工高级 | $100-200/h | 95% | 很慢 | 复杂决策、经验判断 | 数据/格式类复用工作 |
| 人工+LLM混合 | 人工成本-30% | 92% | 快于纯人工 | **几乎所有场景** | — |

### 3.2 成本优化金字塔

```
        ▲
       /人\         $100-200/h    95%准确    终极把关
      /-----\
     / 混合区 \      $10-30/h     92%准确    主力战场（目标）
    /---------\
   /   LLM区  \     $0.01-0.10   85%准确    批量生产
  /------------\
 /   脚本区    \    $0/次        100%准确   无脑自动化（优先）
/--------------\
```

> **黄金法则**：能做脚本的不用LLM，能用LLM的不用人工。

## 四、IB场景映射

### 4.1 场景一：招股书业务与技术章节

```
总任务：撰写业务与技术章节（约30页）
        │
        ├─ [脚本] 整理公司业务数据
        │   - 从审计报告中提取分产品收入/毛利数据
        │   - 自动生成占比/趋势图表
        │
        ├─ [LLM+人工] 行业分析
        │   - LLM搜索+摘录行业报告
        │   - 人工核对数据时效性和权威性
        │   - LLM起草行业描述
        │   - 人工精修、补充公司特色
        │
        ├─ [Lint] 数据一致性
        │   - 确保表格数字与正文描述一致
        │
        └─ [人工] 审阅定稿
            - 整体逻辑流畅度
            - 是否有夸大宣传
            - 是否符合信息披露准则
```

### 4.2 场景二：反馈回复

```
总任务：回复20条问询函
        │
        ├─ [脚本] 问询函拆解
        │   - 自动拆分为子问题列表
        │   - 分配给责任人
        │   - 提取关键法规/条款引用
        │
        ├─ [模板] 75%的标准问询
        │   - 收入确认类→用P01模板
        │   - 毛利率类→用P02模板
        │   - 关联交易类→用P03模板
        │   ……（共8类，见00-META-03）
        │
        ├─ [LLM] 模板初稿生成
        │   - 将公司数据填入对应模板
        │   - 生成可比公司对比段落
        │   - 生成措辞润色
        │
        ├─ [人工] 专项判断
        │   - 25%非标问询（模板不能覆盖的独创性问题）
        │   - 所有涉及主观判断的结论
        │
        └─ [Lint] 最终检查
            - 引用页码是否有效
            - 数据是否与正文一致
```

### 4.3 场景三：尽职调查底稿整理

```
总任务：整理客户/供应商核查底稿
        │
        ├─ [脚本] 底稿索引自动化
        │   - 遍历底稿目录
        │   - 生成索引编号
        │   - 查重检查
        │   - 编号连续性校验
        │
        ├─ [脚本] 数据抽取
        │   - 从Excel走访记录中抽取关键字段
        │   - 自动汇总异常发现
        │
        ├─ [LLM] 底稿描述文字生成
        │   - 生成各循环的底稿说明
        │   - 生成异常事项描述
        │
        └─ [人工] 异常判断
            - 哪些异常需要进一步核查
            - 哪些异常可以在底稿中直接解释
```

### 4.4 场景四：估值模型搭建

```
总任务：搭建DCF估值模型
        │
        ├─ [脚本] 基础数据处理
        │   - 从Wind/审计报告导入历史财务
        │   - 自动计算历史增长率、比率
        │   - 自动敏感性分析（数据表）
        │
        ├─ [人工+LLM] 假设设定
        │   - 人工确定关键假设（增长率、WACC、终值假设）
        │   - LLM搜索可比公司WACC作为参考
        │
        ├─ [脚本] 模型运算
        │   - DCF/可比估值自动计算
        │   - 自动生成敏感性矩阵
        │
        └─ [人工] 合理性校验、区间确定
```

## 五、具体工作流集成示例

### 5.1 示例：一天内完成"关联交易专项分析报告"

```markdown
## 关联交易专项分析（总耗时：3h）

### [脚本] 数据准备 (15min)
python scripts/extract_rpt_data.py --company XXX --output rpt_data.xlsx
→ 自动从审计报告/序时账提取关联交易明细

### [Lint] 完整性检查 (5min)
python scripts/check_rpt_completeness.py --input rpt_data.xlsx
→ 检查关联方是否100%覆盖
→ 检查是否存在交易金额为整数的异常模式
→ 检查定价偏差是否有异常值

### [LLM] 报告起草 (1h)
使用Prompt: "根据以下关联交易数据，生成专项分析报告（含定价公允性、
独立性影响、减少措施三部分），使用标准模板格式。"
→ LLM生成初稿

### [人工] 精修+定稿 (1.5h)
→ 验证每个数据引用的准确性
→ 微调语调和风险披露强度
→ 补充发行人特色信息
→ 确认"减少措施"的可行性和时间线
→ 终稿签字
```

### 5.2 示例：招股书P0→P1跨阶段协作

```markdown
## P0→P1 各工具配合

### [脚本] 全局TBD扫描 (5min)
grep -n "【TBD-" 招股书_v0.1.0.md > tbd_list.txt
→ 自动分类: TBD-数据(XX个) / TBD-确认(XX个) / TBD-撰写(XX个)
→ 按章节自动分组 → 导出到Excel分派表

### [LLM] 批量填充 (2h)
对每个"TBD-撰写"类型的块：
  → LLM根据上下文和框架提示生成填充文本
  
### [Lint] 同步检查 (10min)
→ 填充后的全文交叉引用校验
→ 填充后的表格编号连续性

### [人工] 审定 (3h)
→ 核实在LLM填充的文本
→ 判断各TBD是否真正"已解决"
```

## 六、混合工作的反模式（Anti-Patterns）

| 错误做法 | 为什么错 | 正确做法 |
|---------|---------|---------|
| 用人工做数据搬运和格式调整 | 消耗高成本人力资源做0成本脚本能做的事 | 写到脚本里去 |
| 用脚本试图处理需要语义判断的内容 | 规则写不完，永远有边界情况 | 标记为需要LLM/人工判断 |
| 把整个任务交给LLM，不设人审环节 | 85%正确率对于合规文件是灾难（一页30条，错5条） | LLM必须作为"Draft"而非"Final" |
| 用LLM做纯计算（如"计算三年复合增长率"） | LLM是语言模型，不是计算器 | 用Python/Excel计算 |
| 始终用同一套方法，不根据任务调整 | 浪费或低质量 | 每个任务走一遍决策树 |
| 从不更新模板 | 模板与最新实务脱节 | 每个项目结束后review一次模板 |
| LLM生成后不交叉引用校验 | "数据幻觉"：LLM可能编造数字 | Lint规则跑一遍 |
