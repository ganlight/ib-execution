# 00-META-02 文件迭代工作流

> **适用场景**：IPO招股说明书、再融资募集说明书、重组报告书等核心信息披露文件的全生命周期迭代管理。
> **核心目标**：建立从初稿到注册稿的标准化迭代流程，控制质量梯度和收敛周期，避免"无限迭代"陷阱。

---

## 一、五阶段迭代模型总览

```
P0（初稿）  →  P1（内核稿） →  P2（申报稿） →  P3-P4（反馈稿） →  P5（注册稿）
 3-5天          5-7天          5-7天         各3-5天            3天
 60%完善度      85%完善度      95%完善度      97-99%完善度        100%
```

### 迭代阶段全景图

| 阶段 | 代号 | 典型周期 | 目标完善度 | 核心任务 | 主要工具 |
|------|------|---------|-----------|---------|---------|
| P0 | 初稿 | 3-5天 | 60% | 框架搭建、TBD标记 | 模板库、LLM Draft |
| P1 | 内核稿 | 5-7天 | 85% | 消除TBD、内部交叉审阅 | Review Checklist |
| P2 | 申报稿 | 5-7天 | 95% | 交叉引用校验、格式Lint | Lint脚本 |
| P3-P4 | 反馈稿 | 3-5天×2 | 97-99% | 逐条回复反馈、增量修改 | 模式库（00-META-03） |
| P5 | 注册稿 | 3天 | 100% | 终稿定版、签署盖章 | Final Checklist |

---

## 二、P0：初稿阶段（3-5天，完善度目标60%）

### 2.1 初稿策略

**核心原则**：先完成再完美。初稿的目标不是"好"，而是"全"。

### 2.2 框架搭建流程

```markdown
Day 1 上午: 选模板 + 定章节结构
Day 1 下午: 填写 Risk Factors / Business / MD&A 框架
Day 2:       填写 Financial Information / Related Party / Legal
Day 3:       补充行业分析 / 募集资金运用 / 公司治理
合并+全局审视，标记 TBD
```

### 2.3 TBD标记规范

```markdown
<!-- 使用统一TBD格式，便于后续全局搜索 -->
【TBD-数据】: 待补充数据/数字
【TBD-确认】: 待发行人确认信息
【TBD-撰写】: 待撰写段落（有框架但内容未填充）
【TBD-合规】: 待律师/会计师出具专业意见
【TBD-行业】: 需外部行业数据支撑
```

### 2.4 P0质量基准

- [ ] 全部法定章节已搭建框架（不缺章）
- [ ] 核心财务数据已填入（可能有空值）
- [ ] TBD总数 < 200处（呃...如果超过可能需要增加时间）
- [ ] 无逻辑矛盾（前后章节数据一致）
- [ ] 已嵌入标准风险提示模板

### 2.5 退出条件（Gate P0→P1）

- [ ] TBD清单已完整清点（知道"还差什么"）
- [ ] 各章节责任人均已确认分工
- [ ] 发行人已过目框架结构并确认

---

## 三、P1：内核稿阶段（5-7天，完善度目标85%）

### 3.1 内核稿目标

消解所有TBD，完成内部首次完整审阅，进入"可送内核部门"状态。

### 3.2 内核审阅流程

```
Day 1-3:  逐章消除TBD，填充缺失内容
Day 3-4:  平行拆阅（交叉审阅）：
           - 保荐人A → 审阅业务/行业章节
           - 保荐人B → 审阅财务/法律章节
           - 律师 → 审阅法律相关内容
           - 会计师 → 核对财务数据
Day 5-6:  汇总审阅意见 + 集体讨论
Day 7:    合稿修补 + P1终版
```

### 3.3 P1质量基准

- [ ] TBD已全部消除（实现"零TBD"）
- [ ] 财务数据已与审计报告核对一致（无数字错误）
- [ ] 关键风险因素已差异化（非模板照搬）
- [ ] 关联交易描述已完整（含定价原则、必要性分析）
- [ ] 行业描述已对标可比公司招股书
- [ ] 重大合同/诉讼/行政处罚已覆盖

### 3.4 退出条件（Gate P1→P2）

- [ ] 内核部门反馈意见已全部回应
- [ ] 重大事项（关联交易、同业竞争、历史出资）已深度披露
- [ ] 业务与技术章节与财务数据统计分析一致

---

## 四、P2：申报稿阶段（5-7天，完善度目标95%）

### 4.1 申报稿核心任务

**从"可读"到"无错"的最后一公里。**

### 4.2 交叉引用校验清单

```markdown
## 交叉引用检查（使用 cross_ref_check.sh 脚本自动化）

### 前-后一致性校验
□ 发行人基本情况 → 历史沿革 中股权结构是否一致？
□ 业务与技术 → 财务会计信息 中收入构成是否一致？
□ 关联方清单 → 关联交易 中关联方名称是否一一对应？
□ 募集资金项目 → 业务发展目标 中项目描述是否一致？

### 表-文一致性校验
□ 表格中的数字（金额/比例）与正文叙述是否一致？
□ 表格中的名称（公司名/人名）与正文是否一致？
□ 单位（万元/亿元/%）在各处是否统一标注？

### 外部-内部一致性
□ 审计报告数字 vs 招股书引用数字
□ 法律意见书结论 vs 招股书法律章节
□ 资产评估报告 vs 招股书引用数据
```

### 4.3 Lint规则（自动格式化检查）

```python
# 文件位置: scripts/doc_lint.py
# 自动检查清单

lint_rules = [
    # 格式规范
    ("标题层级连续", check_heading_hierarchy),
    ("表格编号连续无遗漏", check_table_numbering),
    ("页码连续无跳页", check_page_numbers),
    ("章节编号规则一致", check_section_numbering),
    
    # 内容规范
    ("公司简称首次出现带全称括号", check_abbreviation_first_use),
    ("金额数字千分位格式统一", check_number_format),
    ("日期格式统一（YYYY年MM月DD日）", check_date_format),
    ("表格不跨页断裂", check_table_page_break),
    
    # 引用规范
    ("引用页码已全部填入", check_cross_ref_page),
    ("引用章节编号准确", check_cross_ref_section),
    
    # 特殊检查
    ("不存在硬编码的发行人名称错误", check_issuer_name),
    ("不存在占位文本（XXX/TK/TBD/待补充）", check_placeholder),
]
```

### 4.4 P2质量基准

- [ ] 交叉引用检查通过率 ≥ 99%
- [ ] Lint检查零告警（或已确认所有告警为误报）
- [ ] 排版格式与交易所模板完全对齐
- [ ] 签字页信息完整（董监高、保荐人、律师、会计师）

### 4.5 退出条件（Gate P2→申报）

- [ ] 内核部门正式出具《内核批复意见》
- [ ] 全套申报文件（主文件+附属文件）已齐全
- [ ] 质控部门已完成审核签字

## 五、P3-P4：反馈稿阶段（各3-5天）

### 5.1 反馈稿特点

**输入**：交易所/证监会审核问询函  
**输出**：问询回复报告 + 招股书修订版

### 5.2 反馈稿工作流程

```markdown
Day 1 上午: 问询函拆解 → 分配责任人 → 制定回复策略
Day 1 下午: 各责任人同步启动回复撰写
Day 2-3:   回复初稿撰写 + 交叉讨论
Day 4:     内部审阅回复稿 + 修订招股书
Day 5:     提交回复
```

### 5.3 反馈稿质量基准

- [ ] 每条问询均已逐句回应（不遗漏子问题）
- [ ] 回复格式符合要求（先结论再分析，数据支撑）
- [ ] 招股书已同步修订（涉及信息披露补充的均已更新）
- [ ] 回复涉及的核查程序已实际执行
- [ ] 专业意见（律师/会计师）已附具

### 5.4 常见反馈轮次

| 反馈轮次 | P3（首轮） | P4（二轮/落实函） | 额外轮次 |
|---------|----------|-----------------|---------|
| 典型问题数 | 20-40个 | 5-15个 | 0-5个 |
| 回复周期 | 3-5天 | 3-5天 | 2-3天/轮 |
| 重点方向 | 全面覆盖：业务/财务/法律 | 聚焦难点：对首轮回复不满意之处 | 极少数遗留 |

---

## 六、P5：注册稿阶段（3天）

### 6.1 注册稿任务

过会后至注册生效前的最终定稿。重点：招股书更新（如有财务数据更新）+ 风险因素更新（更新至最新风险）+ 募集资金项目更新（如有变化）。

### 6.2 P5 Checklist

```markdown
## 注册稿终审清单

### 财务数据
□ 加期数据已更新（如有）
□ 盈利预测/业绩预告已纳入（如有）
□ 资产负债表日后事项已披露

### 风险因素
□ 最新风险已纳入（如政策变化、市场波动）
□ 重大风险在招股书重大事项提示中列示

### 其他
□ 发行方案已确认（发行股数、发行比例）
□ 募集资金项目无变化（如有变化已更新）
□ 董监高、中介机构签字信息无变更

### 终版确认
□ 发行人确认
□ 保荐人确认
□ 律师确认
□ 会计师确认
```

## 七、收敛准则：何时停止迭代？

### 7.1 强制收敛点

| 阶段 | 最大迭代轮数 | 超过后处理方式 |
|------|------------|---------------|
| P0→P1 | 最多1轮（版本v0.1→v1.0） | 直接进入内核审阅，问题在审阅中解决 |
| P1→P2 | 最多2轮（v1.0→v1.2） | 剩余小问题转入P2并行处理 |
| P2→申报 | 最多1轮 | 内核确认即申报，不追求完美 |
| 反馈回复 | 每轮回复最多1次修订 | 提交后等下一轮反馈再改 |

### 7.2 无限迭代预警信号

🚨 出现以下信号时，立刻叫停并升级：

- [ ] 同一章节被反复修改超过3次（说明方向不清晰）
- [ ] 不同审阅人给出相互矛盾的意见（需要统一标准）
- [ ] 修改量逐轮不减（每轮仍有5%以上内容变动）
- [ ] 新增内容引发新的交叉引用不一致
- [ ] 距计划提交日不足48小时仍在大量修改

### 7.3 收敛决策原则

```markdown
1. 80/20法则：最后20%的完善度消耗80%的时间 → 在95%时强制收束
2. "够用就好"标准：能通过内核/交易所审核即为达标
3. "剩余问题列表"机制：对无法在当轮解决的问题，记录到"下一轮/后续处理"清单
4. 版本冻结原则：每个阶段产出1个冻结版本，后续修改基于冻结版本做增量
```

## 八、迭代模式分类与应对

### 8.1 三种典型迭代模式

| 模式 | 总轮数 | 总周期 | 典型场景 | 特征 |
|------|--------|--------|---------|------|
| **快速收敛型** | 2轮 | 15-20天 | 业务简单、财务清晰、合规良好 | P0→P1→P2 各一轮直通 |
| **标准迭代型** | 3-4轮 | 25-35天 | 存在一定复杂性问题（常见多数项目） | P1/P2 各1-2轮微调 |
| **复杂迭代型** | 5+轮 | 35-50天 | 行业特殊、关联交易复杂、历史遗留问题多 | 需要额外轮次专项处理 |

### 8.2 不同模式的应对策略

**快速收敛型**（Don't over-engineer）：
- 减少冗余审阅环节
- 允许P0跳过完整TBD阶段（直接填内容）
- P1和P2可以合并进行

**标准迭代型**（Follow the process）：
- 严格按五阶段执行
- 每阶段有明确的入场/出场条件

**复杂迭代型**（Divide and conquer）：
- 将复杂问题拆解到独立子任务（如关联交易专项修订）
- 主文件与专项任务并行推进
- 增加专项审阅环节（如第三方专家意见）

## 九、版本管理与痕迹保留

### 9.1 版本编号规则

```
格式: v{major}.{minor}.{patch}
major: 阶段变更（0=P0, 1=P1, 2=P2, 3=P3, 4=P4, 5=P5）
minor: 同阶段内迭代轮次（从0开始）
patch: 小修补（格式调整、错别字修正）

示例:
v0.1.0 = P0阶段第2轮迭代
v2.0.3 = P2阶段第1轮、第4次小修补
```

### 9.2 修订痕迹管理

- **P0→P1**：保留全文修订痕迹（Track Changes）
- **P1→P2**：保留重要章节修订痕迹（业务/财务/法律），格式修改可接受全部
- **反馈回复**：本次修订部分以楷体加粗或下划线标注
- **P5**：清洁版（不保留修改痕迹）

## 十、常见坑

| 易犯错误 | 后果 | 规避方法 |
|---------|------|---------|
| P0追求完善度>80% | 拖延进入内核审阅，浪费P0时间 | P0目标就是60% |
| P1跳过交叉审阅 | 问题到P2才暴露，大返工 | P1必须交叉审阅 |
| 反馈回复不修订招股书 | 下轮反馈重复提问 | 回复同时同步修改招股书 |
| 无限迭代不设收敛点 | 延期申报 | 每个阶段设置硬性收敛点 |
| 版本混乱 | 不知道哪版是最终 | 严格执行版本编号规范 |
| P2检查依赖人工 | 遗漏太多 | 先跑Lint脚本，再做人工 |
| 忽略"剩余问题列表" | 所有问题都要当轮解决 | 允许将非关键问题延后 |
