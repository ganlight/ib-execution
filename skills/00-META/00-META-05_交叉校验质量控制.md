# 00-META-05 交叉校验质量控制

> **版本**: v1.0 | **更新日期**: 2026-05-09 | **适用阶段**: P0~P4 全流程

## 一、核心交叉校验关系

### 1.1 四大核心文件交叉矩阵

| 数据源 | → 招股说明书 | → 审计报告 | → 法律意见书 | → 评估报告 |
|--------|:----------:|:--------:|:----------:|:--------:|
| **招股说明书(P)** | — | P↔A | P↔L | P↔V |
| **审计报告(A)** | A↔P | — | A↔L | A↔V |
| **法律意见书(L)** | L↔P | L↔A | — | L↔V |
| **评估报告(V)** | V↔P | V↔A | V↔L | — |

### 1.2 交叉关系说明

```
招股说明书 ←→ 审计报告
├── 财务数据必须由审计报告支撑
├── 招股书引用的财务数据 = 审计报告审定数
├── 差异不得超过四舍五入误差（±1个最小单位）
└── 审计报告期后事项 → 招股书"重大事项提示"

招股说明书 ←→ 法律意见书
├── 历史沿革描述一致
├── 股东结构、持股比例一致
├── 关联方认定标准及名单一致
└── 诉讼/仲裁披露一致

招股说明书 ←→ 评估报告
├── 评估方法、参数、结论一致
├── 评估基准日一致
├── 评估增减值描述一致
└── 若以评估值入账 → 财务数据必须同步调整

审计报告 ←→ 法律意见书
├── 关联交易金额口径一致
├── 对外担保清单及金额一致
└── 或有负债（未决诉讼等）一致
```

## 二、必须保持一致的8大数据点

| 编号 | 数据点 | 涉及文件 | 校验精度 | 示例 |
|:----:|--------|----------|:--------:|------|
| DP-01 | **净资产** | P, A | 精确到元 | 2023年末归母净资产 850,231,456.78元 |
| DP-02 | **净利润** | P, A | 精确到元 | 2023年度扣非净利润 125,678,901.23元 |
| DP-03 | **营业收入** | P, A | 精确到元 | 2023年度营业总收入 2,345,678,901.23元 |
| DP-04 | **关联交易金额** | P, A, L | 精确到元 | 向关联方采购 50,000,000.00元 |
| DP-05 | **对外担保金额** | A, L, P | 精确到元 | 期末对外担保余额 0 / 200,000,000.00元 |
| DP-06 | **诉讼/仲裁金额** | L, A, P | 精确到元 | 未决诉讼涉案金额 30,000,000.00元 |
| DP-07 | **发行前后持股比例** | P, L | 精确到0.01% | 发行前实控人持股 56.78%，发行后 42.59% |
| DP-08 | **募投项目投资金额** | P | 精确到元 | 总投资 500,000,000.00元 |

### 数据点校验示例

```
场景：DP-04 关联交易金额不一致

招股书 P.256：2023年向母公司的采购金额 52,380,000.00元
审计报告 附注八(二)：2023年向母公司的采购金额 52,380,000.00元   ✅
法律意见书 P.45：2023年关联采购约5238万元                    ⚠️ 精度不一致

→ 处理：统一为精确数字 + 附注中允许约数表述，但正文必须精确
```

## 三、十大 Lint 规则（L-01 ~ L-10）

### L-01: 数据跨文件一致性
```
规则：同一数据点在所有文件中必须完全一致
范围：招股书/审计报告/法律意见书/评估报告/内控报告
触发：任何两份文件中同一数据点差异 > 0.01%
修复：以审计报告为权威源，法律意见书次之
```

### L-02: 术语文件内一致性
```
规则：全文术语必须统一，不得混用同义词
范围：单个文件内部
示例：
  ❌ 文件中混用 "发行人""公司""本公司""股份公司"
  ✅ 律师事务所定义后统一简称
  ❌ 混用 "募集资金""募投资金""募资款"
  ✅ 全文统一使用"募集资金"

修复：建立术语表→全文查找替换→人工复核
```

### L-03: 货币单位统一（万 vs 亿）
```
规则：同一表格/段落中单位必须统一
允许：不同表格可采用不同单位（合并报表万元，明细表元）
禁止：同一表格中出现混用单位
示例：
  ❌ "营业收入 23,456.78万元" + "净利润 1.25亿元"（不同单位混于同段）
  ✅ "营业收入 23,456.78万元，净利润 12,567.89万元"（统一万元）
```

### L-04: 千分位分隔符格式
```
规则：所有 ≥5位 的数字必须使用千分位分隔符
格式：整数部分每3位加逗号
范围：正文、表格、脚注、附录
示例：
  ❌ "2023年营收 2345678901.23元"
  ✅ "2023年营收 2,345,678,901.23元"
```

### L-05: 日期格式标准化
```
规则：全文日期格式统一为"YYYY年MM月DD日"
禁止：中英文混用 / 数字格式混用
示例：
  ❌ "2023年6月30日" / "2023.6.30" / "June 30, 2023"
  ✅ "2023年6月30日"

特殊允许：法律文书中可使用"二〇二三年六月三十日"
```

### L-06: 法规名称准确性
```
规则：引用的法规名称必须与官方公布名称完全一致
包括：文号、发布日期、实施日期、修订状态
示例：
  ❌《公司法》→ 必须标注是否含2018修正内容
  ✅《中华人民共和国公司法》（2018年10月26日修正）

工具：建立法规引用数据库 → 自动校验名称匹配
```

### L-07: 审计期间范围一致性
```
规则：审计报告覆盖期间 = 招股书财务数据期间
标准：最近三年及一期 / 最近三年
边界：若报告期为2019-2022年，则所有引用不得出现报告期外年份的年度数据
例外：可比公司/行业数据的年份引用
```

### L-08: 页码/章节交叉引用
```
规则：文中交叉引用必须指向正确的页码和章节号
触发时机：每次排版后 / 每次增删章节后
示例：
  ❌ "详见本章第三节'同业竞争'"（实际已调整为第四节）
  ✅ "详见本章第四节'同业竞争'"

工具：自动提取所有交叉引用 → 对比目标是否存在 → 标记断裂引用
```

### L-09: 表格/图表编号连续性
```
规则：表格编号必须连续且不重复
格式："表3-2-5" = 第三章第二节第五个表
校验：
  1. 是否存在编号跳跃（表3-2-3 后直接 表3-2-5）
  2. 是否存在编号重复（两个表3-2-4）
  3. 表格号是否与所在章节编号前缀一致
```

### L-10: 签字页完整性
```
规则：各文件中所有需要签字/盖章的页面均已完成
检查项：
  ✅ 发行人法定代表人签字
  ✅ 保荐代表人签字（至少2人）
  ✅ 经办律师签字
  ✅ 经办注册会计师签字
  ✅ 评估师签字
  ✅ 各机构公章
  ✅ 日期填写且一致
  ✅ 骑缝章完整
```

## 四、各阶段质量控制检查点

### P0 阶段（项目启动）→ 轻量检查 L-03 ~ L-10
```
检查目标：确保初稿不犯低级格式错误
检查时机：每份文件初稿完成后
执行方式：
  - L-03 (单位统一): 自动扫描混合单位
  - L-04 (千分位): 自动格式化检查
  - L-05 (日期格式): 正则匹配异常日期
  - L-06 (法规名称): 与新规库比对
  - L-07 (审计期间): 区间一致性自动校验
  - L-08 (交叉引用): 引用目标存在性检查
  - L-09 (编号连续性): 编号序列分析
  - L-10 (签字页): 手动检查清单

不检查：L-01 (数据一致性) / L-02 (术语统一) → 文件未齐备，无法交叉
```

### P1 阶段（文件初汇）→ 加入 L-01 + L-02
```
检查目标：核心文件间数据衔接
检查时机：审计报告初稿 + 法律意见书初稿 + 招股书主要章节完成后
执行方式：
  + L-01: 自动提取各文件的DP-01~DP-08数据点 → 差异报告
  + L-02: 术语表比对 → 标记不一致用法
  + 保留P0阶段的L-03~L-10检查
```

### P2 阶段（内核前）→ 全量 Lint 检查
```
检查目标：提交内核前零差异
检查时机：内核稿定稿前
执行方式：L-01 ~ L-10 全部规则自动运行
通过标准：零告警
特别重点：L-01(数据一致性) 必须0差异
```

### P3+ 阶段（反馈回复/会后事项）→ 增量检查
```
检查目标：每次修改后确保不引入新不一致
检查时机：每次文件修改后（含反馈回复更新）
执行方式：
  - 仅对修改==涉==及的段落重新运行Lint
  - 若修改涉及DP-01~DP-08数据点 → 重新全量交叉校验
  - 若修改涉及章节结构调整 → 重新运行L-08(交叉引用) + L-09(编号连续性)
  - 归档增量差异报告

增量检查流程图：
  修改发生 → 识别影响范围
           ├── 仅文本润色 → 仅L-02(术语) + L-05(日期)
           ├── 财务数据更新 → L-01(全部) + L-03 + L-04
           ├── 章节结构调整 → L-08 + L-09
           └── 法规更新引用 → L-06
```

## 五、cross_ref_check.py 脚本逻辑说明

### 5.1 脚本概述
```
名称：cross_ref_check.py
功能：自动执行 L-01～L-10 的交叉校验
输入：各文件的 Word/PDF（或结构化文本提取结果）
输出：差异报告（HTML / Excel / JSON 三选一）
```

### 5.2 架构模块

```
cross_ref_check.py
├── Module 1: 文档加载器 (DocLoader)
│   ├── 支持格式: .docx, .pdf, .txt
│   ├── 提取: 段落文本, 表格数据, 章节编号, 日期, 数字
│   └── 输出: 统一结构化的 Document 对象
│
├── Module 2: 数据点提取器 (DataExtractor)
│   ├── DP-01~DP-08: 正则 + NLP 提取关键财务数据
│   ├── 上下文感知: 识别"合并口径"/"母公司口径"
│   ├── 单位标准化: 自动统一为"元"
│   └── 输出: {文件: {数据点ID: (值, 页码, 上下文)}}
│
├── Module 3: Lint引擎 (LintEngine)
│   └── 10个独立的 Linter 类:
│       ├── L01Linter: 跨文件数据比对
│       ├── L02Linter: 术语一致性（基于术语表）
│       ├── L03Linter: 单位混用检测
│       ├── L04Linter: 千分位格式检查
│       ├── L05Linter: 日期格式正则
│       ├── L06Linter: 法规名称数据库比对
│       ├── L07Linter: 审计期间交叉验证
│       ├── L08Linter: 交叉引用图遍历
│       ├── L09Linter: 编号序列分析
│       └── L10Linter: 签字页存在性标记
│
├── Module 4: 报告生成器 (Reporter)
│   ├── HTML报告: 交互式差异浏览（按规则/按数据点/按严重程度）
│   ├── Excel报告: 所有差异明细清单
│   ├── JSON报告: 原始差异数据供CI集成
│   └── 严重程度分级: 🔴 阻断 → 🟡 警告 → 🔵 提示
│
└── Module 5: 阶段过滤器 (StageFilter)
    ├── 根据当前项目阶段(P0/P1/P2/P3/P4)选择启用的规则
    ├── P0: L03~L10
    ├── P1: L01~L10
    ├── P2: L01~L10（严格模式，零容忍）
    └── P3+: 增量模式（仅变更区域）
```

### 5.3 核心算法：L-01 数据交叉比对

```python
# 伪代码描述
def check_L01(documents: dict[str, Document]) -> list[Finding]:
    """跨文件数据点一致性检查"""
    findings = []

    # Step 1: 从各文件提取DP-01~DP-08
    data_map = {}
    for filename, doc in documents.items():
        data_map[filename] = extract_datapoints(doc)

    # Step 2: 以审计报告为锚点进行比对
    audit_data = data_map.get("审计报告", {})
    for dp_id in ["DP-01", "DP-02", "DP-03", ...]:
        audit_value = audit_data.get(dp_id)
        if audit_value is None:
            continue
        for filename, file_data in data_map.items():
            if filename == "审计报告":
                continue
            file_value = file_data.get(dp_id)
            if file_value is None:
                findings.append(Finding(
                    level="WARNING",
                    rule="L-01",
                    message=f"{filename} 缺少数据点 {dp_id}"
                ))
            elif abs(audit_value - file_value) > 0.01 * audit_value:
                findings.append(Finding(
                    level="ERROR",
                    rule="L-01",
                    message=f"{dp_id}: 审计报告={audit_value:,} vs {filename}={file_value:,}"
                ))

    # Step 3: 法律意见书 ↔ 招股书（法律数据）
    # 对持股比例、关联方、诉讼金额进行二级比对

    return findings
```

### 5.4 使用示例

```bash
# 全量检查（P2阶段）
python cross_ref_check.py --stage P2 \
  --files 招股书.docx 审计报告.docx 法律意见书.docx 评估报告.docx \
  --output report.html

# 增量检查（P3阶段）
python cross_ref_check.py --stage P3 \
  --changed 招股书.docx  \
  --baseline baseline_snapshot/ \
  --output incremental_report.json

# 仅检查特定规则
python cross_ref_check.py --stage P2 --rules L01,L02,L04 \
  --files 招股书.docx 审计报告.docx
```

### 5.5 CI集成接口

```yaml
# .github/workflows/cross-ref-check.yml
- name: Cross Reference Check
  run: |
    python cross_ref_check.py --stage P2 \
      --files documents/*.docx \
      --output ci_report.json
  # JSON退出码：0=通过，1=发现告警，2=发现阻断错误
```
