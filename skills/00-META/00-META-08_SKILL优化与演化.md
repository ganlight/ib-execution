## 一、核心理念

```
      ┌──────────────────────────────────────────┐
      │                                          │
      │   Skill 不是静态手册，而是活的系统        │
      │                                          │
      │   每个项目 = 一次训练数据采集              │
      │   每次复盘 = 一次参数调优                  │
      │   每次更新 = 一次版本进化                  │
      │                                          │
      │   目标：让下一个项目跑得更快、更准、更稳   │
      │                                          │
      └──────────────────────────────────────────┘
```

### 1.1 演化闭环

```
项目执行 ──→ 数据采集 ──→ 复盘提炼 ──→ 知识入库 ──→ 规则进化
    ▲                                                      │
    └──────────────────────────────────────────────────────┘
                       下一个项目受益
```

## 二、项目后复盘五步法

### Step 1: Retrospective — 3好 + 3改进

```
┌─────────────────────────────────────────────────────────────┐
│ 复盘会议：项目结束后1周内召开                                │
│ 参会人：保荐代表人+项目组成员+质控（可选）                    │
│ 时长：90分钟                                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  📗 三大亮点（Good — 固化下来，形成方法论）                 │
│                                                             │
│  格式模板：                                                 │
│  G1: {亮点描述}                                             │
│      - 做法：我们具体做了什么                                │
│      - 效果：带来了什么价值                                  │
│      - 可复制性：下次能否复用？怎么复用？                    │
│      - 归属：对应用更新哪个Skill文件                        │
│                                                             │
│  实例：                                                     │
│  G1: 招股书"业务与技术"章节采用了可视化图表替代大段文字      │
│      - 做法：券商请来了设计团队，将生产工艺流程用信息图呈现  │
│      - 效果：交易所反馈中表扬"披露清晰易懂"                  │
│      - 可复制性：高。将模板存入 00-META-01 作为推荐格式     │
│      - 归属：更新 00-META-01 § 业务与技术章节模板            │
│                                                             │
│  G2: 二轮反馈回复采用了"问题-分析-结论-索引"四段法          │
│  G3: 中介机构周例会改为每两周一次，效率反而提升              │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  📕 三大改进点（Bad — 分析根因，防止再次发生）              │
│                                                             │
│  格式模板：                                                 │
│  B1: {问题描述}                                             │
│      - 根因：为什么会发生                                    │
│      - 影响：造成了什么后果                                  │
│      - 改进：以后怎么做                                      │
│      - 归属：对应用更新哪个Skill文件                        │
│                                                             │
│  实例：                                                     │
│  B1: 审计报告交付延迟3周，导致整个申报窗口后移              │
│      - 根因：会计师对发行人收入确认方式有异议，反复沟通      │
│              未及时升级到保荐机构协调                        │
│      - 影响：申报时间推迟1个月，错过了Q1审核窗口            │
│      - 改进：在 META-06 中新增规则：                         │
│              重大分歧72小时内必须升级到保荐机构              │
│      - 归属：更新 META-06 § 问题三                           │
│                                                             │
│  B2: 招股书中出现3处章节编号不一致（内部审核发现的）        │
│  B3: 补充2023年年报数据时，关联交易金额未同步更新            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Step 2: Pattern Extraction — 新模式追加到 META-03

```
输入：Step 1 中的 G1~G3、B1~B3
      + 审核过程中交易所/证监会的反馈问题点

处理流程：
  1. 将反馈问题分类归入 META-03 的现有分类
  2. 若出现新类型 → 在 META-03 中新增分类
  3. 填写标准模板：
      - 反馈原文（脱敏）
      - 回复策略
      - 回复模板（脱敏）
      - 适用条件/注意事项
      - 来源项目编号

示例：
  新增模式 → META-03 § 财务类 § 商誉减值测试
  ├── 反馈原文："请发行人补充说明报告期各期末商誉减
  │   值测试的具体过程及参数合理性"
  ├── 回复策略：按《会计监管风险提示第8号》逐项作答
  ├── 回复模板：(见附件模板文件)
  ├── 适用条件：发行人存在重大商誉（>净资产20%）
  └── 来源：项目#2026-003
```

### Step 3: Timeline Correction — 偏差修正 META-07

```
输入：META-07 中的计划时间 vs 本项目实际时间

分析方法：
  1. 计算每个阶段的计划 vs 实际偏差
  2. 分析偏差原因（可归因于基准误差 or 偶发因素）
  3. 对系统性偏差 → 修正基准时长
  4. 对特定条件偏差 → 新增调整因子

示例：
  阶段          计划    实际    偏差    原因                    处理
  ─────────────────────────────────────────────────────────────────
  改制重组      2个月   4个月   +2个月  国资审批超出预期         新增国企因子+1.5月
  辅导验收      4个月   4个月   0       —                       —
  文件制作      3个月   4个月   +1个月  审计延迟（偶发）         不修正基准
  审核问询      4个月   5个月   +1个月  二轮问询新增（较普遍）   修正基准+0.5月
  注册          1个月   1.5月   +0.5月  偶发                    不修正基准
  发行上市      1.5月   1个月   -0.5月  小盘股发行效率高        不修正基准

  执行：
  ✅ 修正 META-07 基准表中"审核问询"时长 4→4.5个月
  ✅ 在 META-07 风险预警中新增"国企改制审批延迟"标记
  ✅ 将偏差数据写入 META-07 § 维护日志
```

### Step 4: Regulatory Updates — 新规同步

```
触发条件：
  - 项目周期内出台的与IPO相关的新法规/政策
  - 交易所更新的审核问答/指引
  - 证监会新发布的监管规定/指导意见

处理流程：
  1. 识别影响范围
  2. 更新所有被引用/影响的Skill文件
  3. 标注更新日期和法规文号

示例：
  2026年3月，证监会发布《关于深化科创板改革
  服务科技创新和新质生产力发展的八条措施》

  影响分析：
  ├── META-01: 科创板招股书模板需新增
  │   "新质生产力"专节
  ├── META-03: 新增"新质生产力"类目反馈模式
  └── META-07: 科创板审核周期可能变化

  执行：
  ✅ 更新 META-01 § 科创板专属章节模板
  ✅ 在 META-03 新增分类 § 新质生产力
  ✅ 在 META-07 添加备注：新政实施后需观察审核周期

跟踪机制：
  建立"法规变更追踪表"

  | 法规名称 | 文号 | 发布时间 | Skill影响文件 | 更新状态 | 更新人 | 日期 |
  |----------|------|:--------:|--------------|:--------:|:------:|:----:|
  | 科创板八条 | 〔2026〕X号 | 2026-03-15 | META-01,03,07 | ✅ 已完成 | 张三 | 03-20 |
  | XXX | X号 | XXXX-XX-XX | META-XX | ⏳ 处理中 | — | — |
```

### Step 5: Archive — 脱敏归档

```
归档内容（脱敏处理）：
  ✅ 项目复盘报告（脱敏版）
  ✅ 新增的回复模板（脱敏）
  ✅ 反馈问答全记录（脱敏）

  ❌ 发行人名称 → 替换为 {公司} / {行业} / {板块}

  ❌ 人名 → 替换为 {保代A} / {律师B}
  ❌ 敏感商业信息 → 概化描述

归档目录结构：
  skills/00-META/
```

## 三、演化节奏

### 3.1 三级更新频率

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  🔄 每项目 → 微更新（Micro Update）                     │
│  ├── 频率：每完成一个IPO项目                            │
│  ├── 内容：                                             │
│  │   - 追加案例到 META-03                               │
│  │   - 记录时间偏差到 META-07                           │
│  │   - 归档项目资料                                      │
│  └── 负责人：项目保荐代表人                            │
│                                                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  🔄 每季度 → 中更新（Medium Update）                    │
│  ├── 频率：每季度末（3月/6月/9月/12月）                │
│  ├── 内容：                                             │
│  │   - 本季度3个以上项目 → 合并分析                    │
│  │   - 模式升/降级（通用↔条件通用↔个案）               │
│  │   - 行业趋势分析                                     │
│  │   - 法规更新追踪                                     │
│  └── 负责人：部门质控 + 行业组负责人                    │
│                                                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  🔄 每年 → 大更新（Major Update）                       │
│  ├── 频率：每年12月                                     │
│  ├── 内容：                                             │
│  │   - 全年所有项目的系统归纳                           │
│  │   - 板块差异矩阵全面更新                             │
│  │   - 重大监管变革的系统适应                           │
│  │   - 跨行业通用模式的提炼                             │
│  │   - Skill文件版本号升级                              │
│  └── 负责人：投行部管理层 + 技术委员会                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 版本号管理

```
所有 META 文件遵循语义版本号：

v{major}.{minor}.{patch}

major: 年度大更新时 +1 (每年12月)
minor: 季度更新时 +1 (结构新增/模式增加)
patch: 项目微更新时 +1 (案例追加/数据修正)

示例：
  META-03 v2.4.7
  2 = 第2年大更新（2026）
  4 = 第4次季度更新（Q4）
  7 = 7次项目微更新
```

## 四、多项目知识合成

### 4.1 模式升级规则

```
┌─────────────────────────────────────────────────────────────┐
│ 出现频次         等级              含义              存储位置│
├─────────────────────────────────────────────────────────────┤
│ ≥3个项目中出现   🥇 通用模式      几乎必出现           META主文件 │
│ 2个项目中出现    🥈 条件通用      特定条件下出现       META主文件+条件标注 │
│ 1个项目中出现    🥉 个案          保留参考价值         archive/ 归档 │
└─────────────────────────────────────────────────────────────┘

升级/降级流程：

个案(1次) ──→ 又出现1次 ──→ 条件通用(2次)
                                 │
                         又出现1次 │ 或连续2个项目未出现
                                 ▼                │
                          通用(≥3次)          降级为个案
```

### 4.2 实战案例

```
模式ID: PATTERN-2025-018
名称: 研发费用资本化论证不足
初始记录: 项目#2025-005（科创板/半导体）- 1次 → 个案
第2次出现: 项目#2025-012（科创板/生物医药）- 2次 → 升级为条件通用
  条件标注: "科创板企业，研发投入>营业收入15%时大概率触发"
第3次出现: 项目#2026-002（科创板/软件）- 3次 → 升级为通用
  执行: 将完整回复策略移入 META-03 § 财务类主文件

降级案例:
  模式: PATTERN-2024-033 "红筹架构拆除中的外汇登记问题"
  出现1次(2024年)，此后连续6个无红筹项目均未触发
  → 年底复查时降级为个案，移入 archive/
```

## 七、自动化支持

### 7.1 skill_evolve.py 脚本逻辑

```
名称：skill_evolve.py
功能：自动化项目复盘的数据采集与知识入库

主要功能：
  1. 输入项目编号 → 自动拉取项目全生命周期数据
  2. 对比计划 vs 实际时间线 → 生成偏差报告 → 建议修正
  3. 自动分类反馈问题 → 比对 META-03 现有模式 → 标记新旧
  4. 分析反馈问题出现频率 → 生成升级建议
  5. 检查法规变更 → 生成影响的Skill文件列表
  6. 生成脱敏版复盘报告草稿
```

### 7.2 使用示例

```bash
# 完整项目复盘
python skill_evolve.py --project PRJ-2026-005 --full

# 仅时间偏差分析
python skill_evolve.py --project PRJ-2026-005 --timeline-only

# 季度批量分析
python skill_evolve.py --quarter 2026Q1 --batch

# 年度全面更新
python skill_evolve.py --year 2026 --major-update
```

## 八、演化健康度检查

### 8.1 健康度指标

| 指标 | 健康值 | 告警值 | 测量方法 |
|------|:------:|:------:|----------|
| 模式命中率 | >60% | <30% | 新项目反馈中能找到对应模式的比例 |
| 时间偏差率 | <15% | >30% | 实际 vs 计划周期的差距 |
| 模式活跃度 | 季度新增 ≥3 | 季度新增 = 0 | 新增/升级模式数量 |
| 法规更新延迟 | <2周 | >1月 | 新规发布到Skill更新的时间差 |
| 跨项目复用率 | >50% | <20% | 复用了历史项目的模板/方案比例 |

### 8.2 改善循环

```
发现健康度告警
  │
  ├── 模式命中率低 → 检查 META-03 分类是否过时
  │    → 重大政策变化后需要中更新甚至大更新
  │
  ├── 时间偏差率高 → 检查 META-07 基准是否需要修正
  │    → 可能是长期趋势变化而非偶发偏差
  │
  ├── 模式活跃度低 → 可能项目量少 or 知识提炼不足
  │    → 即使项目少也要坚持复盘
  │
  └── 法规更新延迟 → 建立法规追踪自动提醒机制
```

---

## 九、总结 — 演化铁律

```
1. 每个项目都要复盘，没有例外
2. 复盘必须产生产出（新模式/修正/归档），不能只是开会
3. "出现3次 = 通用"是硬规则，不能主观判断跳级
4. 脱敏是必须项，不脱敏不上传Skill库
5. 季度更新不可跳过（最晚不要晚于下季度第一周）
6. 外部法规变化 > 内部项目经验（法规更新优先级最高）
```
