# SKILL_04f：文件版本管理

## 一、概述

投行项目的申报文件通常经历数十甚至上百个版本的修改。严谨的版本管理是防止文件混乱、保证数据一致性的基础。本技能定义统一的版本编号规则、文件命名规范、变更追踪要求及协作纪律。

## 二、版本编号规则

### 2.1 五阶段版本体系

| 阶段 | 版本号范围 | 说明 | 典型场景 |
|-----|-----------|------|---------|
| 初稿撰写 | v0.01 ~ v0.99 | 券商项目组内部起草、讨论和修改 | 章节写作、数据填充、内部讨论 |
| 内部审核 | v1.0 ~ v1.99 | 质控部门/内核部门审核修改 | 质控审核意见反馈、内核反馈 |
| 申报提交 | v2.0 ~ v2.99 | 正式向监管机构报送 | 申报稿、补正材料 |
| 反馈回复 | v3.0 ~ v3.99 | 根据监管问询修改 | 一轮/二轮/三轮反馈回复 |
| 注册发行 | v4.0 ~ v5.99 | 注册稿及发行阶段 | v4.x注册稿，v5.x发行稿 |

### 2.2 小版本号含义

- v0.01 ~ v0.09：起草阶段，大块内容填充
- v0.10 ~ v0.49：项目组内部讨论和修改
- v0.50 ~ v0.69：与其他中介沟通后的修改
- v0.70 ~ v0.99：完成内部定稿，提交质控

- v1.0 ~ v1.49：质控反馈后修改
- v1.50 ~ v1.99：内核反馈后修改

- v3.0 ~ v3.49：第一轮反馈
- v3.50 ~ v3.79：第二轮反馈
- v3.80 ~ v3.99：第三轮或补充反馈

### 2.3 版本号升级规则

- **重大改动升级大版本**：如从v0.xx升级至v1.0、从v3.xx升级至v4.0
- **每次保存升级小版本**：如从v0.01→v0.02→v0.03
- **不可回退**：版本号只升不降，回滚需新建分支版本

## 三、文件命名规范

### 3.1 标准命名格式

```
[文件类型]_[项目简称]_[阶段]_v[主版本号].[次版本号].[日期].docx
```

### 3.2 命名示例

```

招股说明书_XX科技_内核稿_v1.30.20260701.docx
招股说明书_XX科技_申报稿_v2.0.20260801.docx
招股说明书_XX科技_反馈回复第一轮_v3.20.20261015.docx
招股说明书_XX科技_上会稿_v3.80.20261201.docx
招股说明书_XX科技_注册稿_v4.0.20270301.docx
招股说明书_XX科技_发行稿_v5.0.20270401.docx

重组报告书_XX重组_草案_v1.0.20260520.docx
```

### 3.3 阶段标识对照

| 阶段 | 标识 |
|-----|------|
| 初稿 | 初稿 / draft |

## 四、变更追踪要求

### 4.1 修订记录维护

每次提交新版文件时，需在文件首页附**修改说明表**：

```
| 版本 | 日期 | 修改人 | 修改内容摘要 | 审核人 |
|------|------|--------|-------------|--------|
| v0.15 | 20260510 | 张某 | 补充行业分析数据 | 李某 |
| v0.16 | 20260512 | 王某 | 修正财务数据口径 | 李某 |
```

### 4.2 报送红线对比

每次正式报送的版本，需提供：
- **红黑修订稿**（Redline/Compare Document）：与上一报送版本的红黑对比
- **修改明细表**：汇总所有修改点及修改理由

### 4.3 Word 修订模式规范

- 所有修改均在Word修订模式下进行（Track Changes = ON）
- 接受修订前，将所有修改保存为一个带修订的版本
- 接受所有修订后另存为新版本号

### 5.2 归档规则

- 所有历史版本均保留，不删除
- 当前工作版本放在最外层或标记为CURRENT
- 每完成一个大阶段，打压缩包归档（含所有工作文件和中间版本）

## 六、并发编辑规则

### 6.1 核心原则：一章一人，一次一章

- **同一章节禁止多人同时编辑**：Word文档无法自动合并多人修改
- **多章可并行**：不同章节不同人可同时推进
- **合并策略**：各章分别完成后由主笔人统一合并为完整文档

### 6.2 多人协作流程

```
1. 主笔人发布章节分工表，指定每章的起草人
2. 各起草人在独立文档中完成各自章节（用模板保持一致格式）
3. 各章节完成后发至主笔人
4. 主笔人合并全文+统稿（统一风格、消除矛盾、检查交叉引用）
5. 统稿后发全体成员审核→修订→定版
```

### 6.3 数据锁机制

对于招股书中的核心数据（如财务数据），设置"数据锁"：
- 财务数据锁定人为项目财务负责人
- 修改财务数据必须经财务负责人确认
- 其他人员发现财务数据问题，不得自行修改，需反馈给财务负责人

## 七、文档格式规范

### 7.1 Word模板要求

- 统一使用项目模板（页眉页脚、封面、目录格式统一）
- 字体：正文宋体小四，标题黑体，英文Times New Roman
- 行距：1.5倍行距
- 表格：表头加粗加灰底，数字右对齐
- 图表：编号统一（图1-1、表1-1等），含清晰标题和数据来源

### 7.2 数字格式规范

| 数据类型 | 格式 | 示例 |
|---------|------|------|
| 金额（万元） | 千分位分隔，两位小数 | 15,280.50 |
| 金额（亿元） | 千分位分隔，四位小数 | 3.5280 |
| 百分比 | 两位小数 | 28.50% |
| 倍数 | 两位小数 | 3.25倍 |
| 数量 | 千分位分隔 | 1,250台 |

## 八、doc_version_diff.py 使用指南

### 8.1 工具功能

`doc_version_diff.py` 是版本对比辅助工具，支持以下功能：

- 生成两个Word文档的红黑对比（输出修订模式的Word文档）
- 生成修改点清单（CSV格式）
- 批量对比目录下所有版本的差异

### 8.2 基本用法

```bash
# 单文件对比
python doc_version_diff.py compare \
  --old "招股说明书_v2.0.20260801.docx" \
  --new "招股说明书_v2.1.20260815.docx" \
  --output "diff_v2.0_v2.1.docx"

# 生成修改清单
python doc_version_diff.py checklist \
  --old "招股说明书_v2.0.20260801.docx" \
  --new "招股说明书_v2.1.20260815.docx" \
  --output "修改清单_v2.0_v2.1.csv"

# 批量对比
python doc_version_diff.py batch \
  --dir "./招股说明书/" \
  --pattern "v2.*.docx" \
  --output "./对比结果/"
```

### 8.3 输出文件说明

- 红黑对比.docx：带修订模式的Word文档，红色=新增，黑色删除线=删除
- 修改清单.csv：列包括 页码、修改类型（新增/删除/修改）、修改前文本、修改后文本、修改人、修改日期
